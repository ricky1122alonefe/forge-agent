## Reporting a Vulnerability

**Please do not report security vulnerabilities through public GitHub issues.**

Instead, please report them via one of the following methods:

### Option 1: GitHub Security Advisories (preferred)

Open a [private security advisory](https://github.com/ricky1122alonefe/forge-agent/security/advisories/new)
on GitHub. This allows us to collaborate on a fix before public disclosure.

### Option 2: Email

Send to [security@forge-agent.example.com](mailto:security@forge-agent.example.com)
with the subject line starting with `[SECURITY]`.

You should receive a response within **48 hours**. If for some reason you
do not, please follow up via GitHub Security Advisories.

---

## What to Include

Please include as much of the following information as possible:

- **Type of issue** (e.g. buffer overflow, SQL injection, cross-site scripting)
- **Full paths of source file(s)** related to the issue
- **Location of the affected source code** (tag/branch/commit or direct URL)
- **Step-by-step instructions to reproduce** the issue
- **Proof-of-concept or exploit code** (if possible)
- **Impact** of the issue, including how an attacker might exploit it

This information will help us triage your report more quickly.

---

## Our Commitment

When you report a vulnerability:

1. **We will acknowledge** receipt of your report within 48 hours
2. **We will keep you informed** of the progress toward a fix
3. **We will credit you** in the fix release (unless you prefer to remain anonymous)
4. **We will coordinate** the disclosure timeline with you

We aim to:

- **Triage** within 3 business days
- **Provide a fix** for critical issues within 30 days
- **Release a security advisory** with full credit to the reporter

## Security Considerations Specific to forge-agent

`forge-agent` is a **code-generation framework**. This means:

### 1. Generated code is untrusted

Code generated by the LLM is executed in the user's environment. Users are
responsible for:

- **Reviewing generated code** before activating
- **Running in a sandboxed environment** for untrusted requirements
- **Verifying imports** in generated code (the validator helps, but is not
  a substitute for human review)

The `Sandbox` class in `forge_agent.generator.sandbox` provides resource
isolation but is **not a security boundary**. See the
[Sandbox docs](docs/architecture/sandbox.md) for details.

### 2. LLM API keys

`forge-agent` does **NOT** store or transmit your LLM API keys to any
third party. Keys are read from:

1. Environment variables (e.g. `OPENAI_API_KEY`)
2. The system keyring (via `keyring` library)
3. A local `llm_providers.json` config file (you create this)

**Never commit API keys to git.** The `.gitignore` includes
`llm_providers.json` and `*.local.json` for this reason.

### 3. MCP (Model Context Protocol)

When connecting to MCP servers, only connect to **trusted servers**.
forge-agent will surface the available tools but does not restrict what
the generated code can do with them.

### 4. Dynamic code execution

The `injector` module dynamically imports generated code. This is
**inherently dangerous** if the generated code comes from an untrusted
source. The validator reduces but does not eliminate this risk.

## Security Updates

Security fixes are released as patch versions (e.g. `0.3.1`) and
announced in:

- [GitHub Security Advisories](https://github.com/ricky1122alonefe/forge-agent/security/advisories)
- [CHANGELOG.md](CHANGELOG.md) under the "Security" section
- GitHub Releases (with the `security` label)

We recommend pinning to specific minor versions in production:

```toml
# pyproject.toml
dependencies = [
    "forge-agent>=0.3,<0.4",  # Accept patches and minors, no breaking
]
```
